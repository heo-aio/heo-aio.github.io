데이터 하나가 NumPy 배열이 되었다가, Pandas 표 위에 얹혀 가공되고, 마지막에 Matplotlib으로 눈에 보이는 그림이 되는 순서다.

# 1️⃣ NumPy — 결국 shape와 axis 싸움이다

NumPy 예제 70문제를 풀었다.

앞쪽 기초는 파이썬 리스트와 비슷해서 무리 없이 넘어갔지만, 중간에 크게 걸린 개념이 몇 가지 있었다.

## 뷰(view)와 복사(copy)

슬라이스가 원본과 메모리를 **공유**한다는 점을 처음 보고 당황했다.

파이썬 리스트 감각으로는 `b = a[:3]`이 당연히 새로운 배열을 복사해 온 것이라고 생각했는데, NumPy는 그렇지 않았다.

```python
a = np.array([1, 2, 3, 4, 5])
b = a[:3]     # 뷰(view) — 원본과 메모리 공유
b[0] = 99
print(a)      # [99  2  3  4  5]  ← 원본이 함께 바뀐다
```

`b`만 수정했는데 `a`가 바뀌어 있어서 한참 코드를 다시 봤다.

슬라이스는 기본적으로 뷰이기 때문에, 값을 진짜로 복사하려면 `.copy()`를 명시적으로 호출해야 한다.

전처리 코드에서 이걸 모르고 원본을 건드리면 찾기 힘든 버그가 될 수 있겠다는 생각이 들었다.

> **핵심 정리**
>
> 슬라이스는 기본적으로 뷰(view)이며 원본과 메모리를 공유한다. 원본을 보존하려면 반드시 `.copy()`로 복사해야 한다.

## axis — 사라지는 축으로 이해하기

집계 함수의 `axis` 인자가 계속 헷갈렸다.

`sum(axis=0)`이 열 방향인지 행 방향인지 매번 헷갈려서, 결국 **"axis는 사라지는 축"** 이라고 이해하기로 했다.

```python
a = np.array([[1, 2],
              [3, 4],
              [5, 6]])   # shape (3, 2)

a.sum(axis=0)   # [ 9 12 ]     ← 0번 축(행)이 사라짐 → 열별 합
a.sum(axis=1)   # [ 3  7 11 ]  ← 1번 축(열)이 사라짐 → 행별 합
```

"어느 축이 없어지는가"로 생각하니 그제서야 헷갈리지 않았다.

## 브로드캐스팅과 벡터화

모양이 다른 배열끼리 자동으로 맞춰 계산해 주는 브로드캐스팅은, 규칙을 알고 나니 오히려 코드가 짧아져서 좋았다.

반복문을 돌리지 않고 배열을 통째로 계산하는 것이 벡터화의 핵심인데, 예를 들어 평균 제곱 오차(MSE)도 반복문 없이 한 줄로 끝난다.

$$
MSE = \frac{1}{n} \sum_{i=1}^{n} (\hat{y}_i - y_i)^2
$$

```python
mse = np.mean((y_pred - y_true) ** 2)
```

이 한 줄을 보고 나서, 왜 다들 반복문 대신 벡터 연산을 쓰라고 하는지 몸으로 이해했다.

## 정형 데이터 가공 (61~70번)

마지막 파트가 가장 실무에 가까웠다.

서버 점검 CSV를 한 번만 `read_csv`로 읽어 `to_numpy()`로 바꾼 뒤, 그다음부터는 순수 NumPy만으로 가공하는 문제였다.

특히 열 방향 표준화(z-score)를 직접 계산하는 부분이 기억에 남는다.

$$
z = \frac{x - \mu}{\sigma}
$$

```python
mean = X.mean(axis=0)
std  = X.std(axis=0)
X_std = (X - mean) / std     # 열 단위 표준화
```

이 외에도 `np.argsort`로 조치 시간 상위 뽑기, `np.unique(..., return_inverse=True)`로 범주를 정수 라벨로 인코딩하기, `np.random.permutation`으로 섞어서 train/valid 나누기 같은 작업을 다뤘다.

sklearn 없이도 라벨 인코딩과 표준화가 원리대로 되는 것을 직접 본 느낌이라 좋았다.

> **핵심 정리**
>
> NumPy만으로도 표준화, 라벨 인코딩, train/valid 분할 같은 전처리의 원리를 구현할 수 있다. 라이브러리에 의존하기 전에 원리를 이해하는 것이 중요하다.

# 2️⃣ Pandas — 실제 데이터 세 개로 감을 잡다

Pandas는 서버 점검, 군수품 청구, 장병 복지시설 세 가지 CSV를 놓고 65문제를 풀었다.

예제용으로 깔끔하게 정리된 데이터가 아니라 컬럼도 많고 결측치도 있는 데이터라 훨씬 실전에 가까웠다.

## loc와 iloc — 라벨이냐 위치냐

가장 기본이면서 가장 자주 헷갈린 것이 `loc`과 `iloc`이었다.

정리하면 `loc`은 **이름(라벨)** 기반, `iloc`은 **정수 위치** 기반이다.

```python
df.loc[:, ["server_id", "cpu_usage"]]   # 열 '이름'으로 선택
df.iloc[:5, :3]                          # 앞 5행, 앞 3열 (위치)
```

헷갈릴 때는 "loc은 label의 L"이라고 외우기로 했다.

## 조건 필터 — and 대신 &

복합 조건 필터에서 파이썬 습관대로 `and`를 썼다가 에러가 났다.

Pandas에서는 `&`, `|`를 쓰고, 각 조건을 **괄호로 감싸야** 한다.

```python
# 틀린 코드: df[df["fix_required"] == 1 and df["issue_detected"] == 1]  → 에러
df[(df["fix_required"] == 1) & (df["issue_detected"] == 1)]   # 올바른 코드
```

여러 값 중 하나면 되는 조건은 `isin([...])`으로, 두 값 사이 구간은 `between(60, 80)`으로 쓰니 코드가 훨씬 깔끔해졌다.

## groupby — 쪼개고, 계산하고, 합친다

`groupby`는 "그룹으로 쪼개고, 각 그룹에서 계산하고, 다시 합친다"는 흐름으로 이해하니 결과가 납득됐다.

```python
df_srv.groupby("issue_category")["cpu_usage"].mean()
df_srv.groupby("check_type")["fix_duration_hours"].agg(["mean", "max"])
```

`agg`로 한 번에 여러 통계를 뽑는 것, `size()`는 결측과 무관한 그룹 행 수이고 `count()`는 비결측만 센다는 미묘한 차이도 함께 정리했다.

## merge — inner인데 행이 늘어났다

군수품과 복지 데이터를 `unit_code`로 inner merge 했는데, 결과 행 수가 원본보다 많아져서 당황했다.

같은 키가 양쪽에 여러 개 있으면 조합만큼 행이 곱해진다(다대다).

"merge하면 무조건 행이 줄어든다"고 막연히 생각했던 것이 틀렸던 것이다.

> **핵심 정리**
>
> merge는 항상 행을 줄이는 연산이 아니다. 키가 양쪽에서 유일하지 않으면 행이 오히려 늘어날 수 있으므로, merge 전에 키의 유일성을 확인하는 습관이 필요하다.

## 시계열 — 일단 datetime으로

날짜가 문자열(`object`)로 들어와 있으면 아무 시계열 연산도 할 수 없다.

`pd.to_datetime`으로 변환한 뒤에야 `.dt.year`, `.dt.month`, `.dt.dayofweek` 같은 `.dt` 접근자를 쓸 수 있다.

```python
df["check_date"] = pd.to_datetime(df["check_date"], errors="coerce")
df["check_date"].dt.year
```

깨진 값은 `errors="coerce"`로 NaT 처리한다는 점, 그리고 `dayofweek`는 월요일이 0이고 일요일이 6이라는 점을 따로 메모해 두었다.

# 3️⃣ Matplotlib — fig, ax를 써야 하는 이유

솔직히 이번 공부에서 가장 크게 얻은 것은 이 부분이다.

그동안 아무 생각 없이 `plt.plot()`, `plt.title()`을 써왔는데, 왜 잘 짠 예제는 모두 `fig, ax = plt.subplots()`로 시작하는지 이번에 이유를 알았다.

## plt 직접 호출 vs 객체지향 방식

두 방식은 결과 그림은 같지만 성격이 완전히 다르다.

```python
# 방식 1 — plt 직접 호출 (암묵적)
plt.figure(figsize=(6, 4))
plt.plot([1, 2, 3], [4, 5, 6])
plt.title("title")

# 방식 2 — fig, ax 명시적 (객체지향)
fig, ax = plt.subplots(figsize=(6, 4))
ax.plot([1, 2, 3], [4, 5, 6])
ax.set_title("title")
```

### 📷 실행 결과

![fig, ax 객체지향 방식 예시](/images/data/matplotlib_fig_ax.png)

## plt를 직접 쓰면 안 되는 이유

핵심은 `plt`가 **"현재 활성화된 그림"을 전역 상태로 관리**한다는 점이다.

`plt.plot()`은 "지금 활성화된 어딘가"에 그리는 것인데, 그 대상이 코드 흐름에 따라 오류도 경고도 없이 조용히 바뀐다.

특히 반복문이나 함수 안에서 `plt.subplot()`을 쓰면, 의도한 axes가 아니라 "그 순간 활성화된" axes에 그려진다.

버그가 코드 에러가 아니라 그림 결과로만 나타나기 때문에 원인을 찾기가 매우 어렵다.

반면 `fig, ax`로 객체를 직접 쥐고 있으면 `axes[0]`, `axes[1]`처럼 원하는 subplot을 콕 집어 제어할 수 있다.

`fig.suptitle()`, `fig.tight_layout()`처럼 그림 전체를 제어하는 메서드도 `fig` 객체가 있어야 호출할 수 있다.

## seaborn을 쓸 거면 사실상 필수

결정적으로 seaborn은 모든 함수가 `ax=` 인자로 그릴 대상을 받는다.

`ax=`를 지정하지 않으면 seaborn이 새 그림을 만들거나 엉뚱한 axes에 그려서 기존 subplot 레이아웃이 망가진다.

앞으로 seaborn을 자주 쓸 것 같은데, 그렇다면 `fig, ax` 방식은 선택이 아니라 기본으로 깔고 가야겠다고 생각했다.

> **핵심 정리**
>
> `plt` 직접 호출은 전역 상태에 의존하기 때문에 실행 순서에 따라 결과가 달라진다. 재현성과 subplot 제어를 위해 `fig, ax` 객체지향 방식을 기본으로 쓰는 것이 좋다.

처음에는 이 세 라이브러리를 그냥 "문법을 외우는 것"이라고 생각했다.

하지만 예제를 직접 풀어 보니 중요한 것은 함수 이름을 아는 것이 아니라, **동작은 하지만 왜 그렇게 되는지 몰랐던 부분을 이유까지 이해하는 것**이라는 점을 알게 되었다.

AI 서비스를 만든다는 것은 결국 이런 전처리와 데이터 확인 위에서 모델이 돌아가는 것이니, 이 기본기를 대충 넘기지 않은 것이 나중에 도움이 될 것이라고 생각한다.
